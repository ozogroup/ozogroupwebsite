# KIA Skin Care Business Engine Audit

Date: 2026-07-14
Branch: `codex/kia-production-launch-repair`
Production deploy: not performed

## Executive Audit

The restored helper patch gives the app the required ID and approval foundation: membership IDs, booking IDs, treatment order IDs, partner-code generation, referral lookup, non-recursive admin RLS, and atomic paid-membership approval are present in the generated type map and app code.

The main launch blocker found in this pass was financial attribution. Booking commissions were generated with the booking's direct/referral partner as Level 1 earner. The final business model requires the booking source partner's sponsor to be Level 1, then the sponsor chain above them as Levels 2, 3, and 4. The app also used `partner_sales` for some dashboard/report sales totals. That table is now treated as compatibility metadata only; business totals read from `bookings`, `commissions`, `payouts`, and `wallet_transactions`.

## Launch Blockers

### BLOCKER

- Booking commission hierarchy did not match the final 4-level model.
  - Previous behavior: source partner earned Level 1.
  - Required behavior: source partner's sponsor earns Level 1, sponsor's sponsor earns Level 2, up to Level 4 only.
  - Fixed in app fallback and prepared DB RPC migration: `public.kia_generate_booking_commissions(uuid)`.

### CRITICAL

- Admin Commissions page did not show source booking, source amount, rate, payout link, or paid date.
  - Fixed by enriching commission rows with booking details and updating the admin table.

- Partner/admin dashboards used `partner_sales` for sales/report totals.
  - Fixed dashboard/report reads to use `bookings`.

### HIGH

- `partner_sales.commission_amount` was populated with an estimated direct commission during booking creation.
  - Fixed future writes to store `null` so it cannot be mistaken for money truth.

- Paid payout marking did not stamp `commissions.paid_at`.
  - Fixed `markApprovedCommissionsPaid` to set `paid_at`.

### MEDIUM

- Partner commission page still had legacy `commission_amount` fallbacks and broken rupee encoding.
  - Fixed to read `commissions.amount` only and display `Rs.`.

### LOW

- `partner_sales` still receives non-financial booking mirror rows for backward compatibility.
  - Not used for current admin/partner money totals.

## Admin Panel Audit

- Dashboard: loads counts and recent records. Financial cards now use `bookings`, `commissions`, `payouts`, and `partners`, not `partner_sales`.
- Membership Requests: uses `memberships`; approval uses `kia_approve_paid_membership`. Pending, paid, active, and rejected rows remain visible.
- Bookings: create/update status/payment flows call server actions; paid + confirmed/completed invokes commission generation. Booking IDs and treatment order IDs are DB generated by existing patch triggers.
- Commissions: now traceable by partner, source booking, level, rate, source amount, commission amount, status, created date, paid date, payout ID.
- Payouts: reads real payout rows plus available wallet summaries. Paid payout flow marks linked approved commissions as paid and stamps `paid_at`.
- Reports: export-ready data now reads bookings, memberships, commissions, payouts, partners. Commission export includes `source_id`, `source_amount`, `payout_id`, `paid_at`.
- Contact Settings: already uses admin-guarded server action and service client. No code changes in this pass.
- Treatments/Testimonial/FAQ/Content: CRUD server actions present from earlier work; not redesigned in this pass.
- KYC: review actions present; payout still requires verified KYC and bank verification.
- Franchise Leads/Payments/Settings/System Health: loaded and scoped by existing actions; no confirmed launch-blocking bug found in this pass.

- Membership ID is DB generated at insert by the restored patch.
- Approval is atomic through `kia_approve_paid_membership`: profile role, membership status, partner status, partner code, referral tree linkage.

- Partner dashboard now uses real bookings for own sales/top kits/recent bookings and real commissions for income.
- Partner direct team and my team pages preserve pending and approved sponsored members.
- Partner payout requires active partner, verified KYC, verified bank, and minimum wallet balance.

## Referral Audit

- Canonical code column: `partners.partner_code`.
- Secure lookup helper exists: `kia_lookup_referrer(text)`.
- Referral tree supports Levels 1-4.
- Prepared commission RPC walks sponsor chain with cycle prevention and stops at Level 4.
- No Level 5 generation is present in the app fallback or migration.

## Booking Audit

- Booking records include customer, treatment, amount, referral/partner code, payment status, booking status, generated booking ID, generated treatment order ID.
- Booking create still writes a compatibility row to `partner_sales`, but no commission amount is stored there and app totals no longer depend on it.
- Paid + confirmed/completed booking triggers commission ledger generation.

## Commission Audit

- One ledger table: `commissions`.
- Final rates: 6%, 3%, 1.7%, 1.2%.
- Prepared migration adds partial unique index for idempotency:
  `source_type + source_id + partner_id + level WHERE deleted_at IS NULL`.
- App fallback also checks existing commission rows before insert.
- Wallet is credited only when admin approves a pending commission.
- Paid state is applied when payout is marked paid.

## Wallet Audit

- Wallet increases only in `approveCommission`.
- Wallet decreases only in payout paid flow/RPC fallback.
- Wallet transactions are written for commission credit, commission reversal, and payout debit.
- Partner dashboard wallet reads `partners.wallet_balance`.

## Payout Audit

- Partner request calculates gross, 15% deduction, net payable.
- Admin payout page shows gross, deduction, net, wallet/paid, bank/UPI, status, and action controls.
- Mark-paid lifecycle now updates payout status, wallet, wallet transaction, linked commissions, and `paid_at`.

## Financial Ledger Design

Source of truth:

1. Revenue: `bookings.payment_amount`, `memberships.payment_amount`.
2. Commission liability: `commissions` rows.
3. Wallet liability: `partners.wallet_balance` plus `wallet_transactions`.
4. Payout settlement: `payouts` plus `commissions.payout_id`.

Booking by Partner E:

1. Source partner = E.
2. Level 1 earner = E's sponsor D at 6%.
3. Level 2 earner = C at 3%.
4. Level 3 earner = B at 1.7%.
5. Level 4 earner = A at 1.2%.
6. Stop.
